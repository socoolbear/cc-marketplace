# 지식 아키텍처 패턴

프로젝트 지식을 "지도 + 시스템 오브 레코드" 구조로 편성하는 패턴.
에이전트가 최소한의 컨텍스트로 정확한 정보에 접근할 수 있도록 설계한다.

---

## 1. 지도 원칙 — AGENTS.md 는 ~100줄 포인터 모음

### 왜 지도인가

OpenAI 의 교훈: **"에이전트에게 1000페이지 매뉴얼이 아니라 지도를 줘라."**

에이전트는 사람과 다르다. 사람은 긴 매뉴얼을 선택적으로 읽지만,
에이전트는 주어진 컨텍스트를 전부 처리해야 한다.
진입점이 가볍고, 필요할 때 심화 문서를 참조하는 구조가 최적이다.

### 거대한 지시 파일의 문제

| 문제 | 설명 |
|------|------|
| **컨텍스트 자원 소모** | 500줄짜리 AGENTS.md 는 매 에이전트 턴마다 토큰을 소비한다. 대부분 해당 작업에 불필요한 정보다. |
| **과도한 가이드는 비가이드** | 모든 걸 지시하면 아무것도 지시하지 않은 것과 같다. 에이전트는 우선순위를 판단하지 못한다. |
| **즉시 부패** | 파일이 길수록 일부 섹션이 코드와 어긋날 확률이 높아진다. 에이전트가 낡은 정보를 따르면 오류가 발생한다. |
| **검증 어려움** | 긴 파일은 리뷰가 힘들다. 어떤 규칙이 유효한지, 어떤 규칙이 사라져야 하는지 판단이 모호해진다. |

### 점진적 노출 (Progressive Disclosure)

```
AGENTS.md (지도, ~100줄)
  ├─ 프로젝트 개요 (5줄)
  ├─ 명령어 (빌드, 테스트, 린트)
  ├─ 아키텍처 요약 → docs/architecture.md
  ├─ Phase 테이블 → docs/phases/phase-N-*.md
  ├─ 교훈 요약 → docs/references/*.md
  ├─ 도구 → scripts/
  └─ 문서 지도 (전체 인덱스)
```

**원칙:**
- AGENTS.md 에는 **규칙 자체를 쓰지 않는다.** 규칙이 어디에 있는지를 쓴다.
- 각 포인터는 한 줄로: `아키텍처 규칙 → docs/architecture.md`
- 유일한 예외: 빌드/테스트/린트 명령어는 AGENTS.md 에 직접 기록 (가장 자주 참조되므로)

## 2. docs/ 구조 템플릿

### 표준 디렉터리 구조

```
docs/
  architecture.md     # 아키텍처 규칙의 권위적 원천 (단일 파일)
  phases/             # Phase 별 스펙 + 성공 기준
    phase-0-foundation.md
    phase-1-core.md
    phase-2-features.md
    ...
  references/         # 핵심 레퍼런스 (실패 교훈, 타입 계약, 도메인 지식)
    failure-lessons.md
    type-contracts.md
    domain-glossary.md
    ...
  quality/            # 품질 점수 + 시간순 로그
    scores.json
    quality-log.md
  legacy-*/           # 동결된 기존 문서 (읽기 전용)
    legacy-architecture/
    legacy-readme/
```

### 각 디렉터리 역할

| 디렉터리 | 역할 | 갱신 주기 |
|-----------|------|-----------|
| `architecture.md` | 아키텍처 불변 조건 정의. 다른 모든 문서보다 우선. | 구조 변경 시 |
| `phases/` | 각 Phase 의 범위, 산출물, 성공 기준. Phase 시작 전 확정. | Phase 시작/종료 시 |
| `references/` | 반복 참조되는 지식. 실패에서 배운 교훈, 타입 계약 등. | 교훈 발생 시 |
| `quality/` | 기계 판독 가능한 점수 + 사람이 읽는 로그. | Phase 평가 완료 시 |
| `legacy-*/` | 마이그레이션 이전 문서. 수정 금지. | 절대 갱신하지 않음 |

## 3. architecture.md 작성 가이드

하나의 파일에 프로젝트의 아키텍처 불변 조건을 모두 담는다.
**이 파일이 아키텍처에 관한 유일한 권위적 원천 (Single Source of Truth) 이다.**

### 필수 섹션

```markdown
# 아키텍처

## 레이어 구조
(레이어 이름, 책임, 허용 import 방향을 DAG 으로)

## 타입 경계
(외부 입력 타입, 내부 상태 타입, 저장 포맷 타입의 관계)

## 의존성 정책
(허용 패키지 vs 금지 패키지, 패키지 추가 절차)

## 파일 컨벤션
(네이밍 규칙, 디렉터리 배치, 배럴 export 정책)

## 빌드 및 배포
(빌드 스크립트, 환경 변수, 배포 프로세스)
```

### 작성 규칙

- 각 규칙에는 **이유** 를 한 줄 이상 포함한다 (에이전트가 규칙의 의도를 이해해야 올바르게 적용)
- 예시 코드는 최소화한다 (예시는 부패하기 쉬움 — 규칙만 명확히)
- 모호한 표현 금지: "가능하면", "일반적으로" 대신 "반드시", "금지" 사용

## 4. Phase 스펙 작성 가이드

각 Phase 는 `docs/phases/phase-N-이름.md` 파일로 독립 관리한다.

### 필수 섹션

```markdown
# Phase N: [이름]

## 목표
(1-2줄로 이 Phase 가 달성할 것)

## 산출물
(생성/수정할 구체적인 파일 또는 함수 목록)

## 성공 기준
(측정 가능한 체크리스트. "~이 동작한다" 형태)
- [ ] 타입 에러 0
- [ ] 모든 테스트 통과
- [ ] 레이어 위반 0
- [ ] ...

## 위험 요소
(이전 Phase 의 실패 교훈 또는 예상 위험)
→ docs/references/failure-lessons.md 연결

## 의존성
(이전 Phase 결과물 중 이 Phase 가 전제하는 것)

## 레이어 범위
(이 Phase 가 건드리는 레이어 목록)
```

### 주의

- **성공 기준은 자동 검증 가능하게** 작성한다 (Evaluator 가 판단할 수 있어야)
- "~하지 않을 것" 형태의 **부정 기준** 도 포함한다 (회귀 방지)

## 5. 레퍼런스 추출 기준

다음 중 하나에 해당하면 `docs/references/` 에 별도 문서로 추출한다:

| 기준 | 예시 |
|------|------|
| **이전 실패 교훈** | "core 에서 React 를 import 하면 순환 의존 발생" |
| **도메인 지식** | 게임 데이터의 스탯 계산 공식, API 응답 형태 |
| **타입 계약** | 외부 API 타입 → 내부 타입 변환 규칙 |
| **3회 이상 반복 참조** | 여러 Phase 스펙에서 동일 정보를 참조하면 추출 |

### 추출 문서 작성 규칙

- 자기완결적: 다른 문서를 읽지 않아도 이 문서만으로 해당 주제를 이해할 수 있을 것
- 형식: 증상 → 원인 → 해결 규칙 (실패 교훈의 경우)
- 최상단에 한 줄 요약 포함

## 6. 문서 동결 정책

기존 프로젝트의 문서를 하네스 구조로 마이그레이션할 때,
원본 문서는 **동결** 하여 `docs/legacy-*/` 에 보관한다.

### 동결 절차

1. 원본 문서를 `docs/legacy-원본위치/` 로 이동
2. 파일 최상단에 동결 공지 추가:
   ```markdown
   > **[동결]** 이 문서는 YYYY-MM-DD 에 동결되었습니다.
   > 최신 정보는 `docs/architecture.md` 를 참조하세요.
   > 이 파일을 수정하지 마세요.
   ```
3. AGENTS.md 의 문서 지도에서 legacy 경로를 명시

### 이유

- 기존 문서에 담긴 맥락이 유실되지 않도록 보존
- 새 구조(docs/)와 혼동되지 않도록 격리
- 동결 공지가 에이전트에게 "이 정보는 오래된 것"임을 알림
